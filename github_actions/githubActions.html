GitHub Actions Overview

GitHub Actions is a continuous integration and continuous delivery (CI/CD) platform integrated directly into GitHub. It enables the automation of software development workflows, including building, testing, and deploying code.

Key Concepts:

  • Workflows: Automated processes defined in YAML files within the .github/workflows directory of a repository. They are triggered by specific events, such as a push to a branch, a pull request, or a scheduled time.
  • Events: Actions are triggered by events occurring in a repository. Examples include push, pull_request, issue_comment, or schedule.
  • Jobs: A set of steps that execute on the same runner (a virtual machine or container). Jobs can run in parallel or sequentially and can have dependencies on other jobs.
  • Steps: Individual tasks within a job. A step can be a shell script or an action.
  • Actions: Reusable units of code that perform a specific task. They can be custom-created, found on the GitHub Marketplace, or provided by GitHub. Actions can be written in JavaScript or packaged as Docker containers.
  • Runners: The environments where jobs execute. GitHub-hosted runners provide a variety of operating systems and software, while self-hosted runners allow for custom environments.

Job Execution and Dependencies

By default, if you define multiple jobs in a GitHub Actions workflow, and they do not have explicit dependencies on each other using the needs keyword, they will run in parallel.

Each job will execute on its own runner (either GitHub-hosted or self-hosted) and independently of other jobs in the workflow.
